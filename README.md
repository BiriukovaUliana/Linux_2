# Практична робота №2

### Завдання 2.1
Напишіть програму для визначення моменту, коли time_t закінчиться. Дослідіть, які зміни відбуваються в залежності від 32- та 64-бітної архітектури. Дослідіть сегменти виконуваного файлу.

### Обгрунтування

Написано програму, яка визначає розмір time_t на поточній архітектурі, обчислює максимально можливе значення за допомогою INT32_MAX або INT64_MAX, конвертує його у дату через функцію gmtime() та демонструє ефект переповнення.
Програму скомпільовано у двох варіантах за допомогою компілятора GCC. Для 64-бітної версії використано стандартну команду gcc -Wall -o task2_1_64 task2_1.c, для 32-бітної — прапор -m32: gcc -Wall -m32 -o task2_1_32 task2_1.c.

### Результат роботи програми

При запуску 64-бітної версії програма показала, що time_t займає 8 байтів і переповнення відбудеться приблизно у 292 277 026 596 році. При запуску 32-бітної версії — time_t займає 4 байти і переповнення станеться 19 січня 2038 року.

- Стек росте донизу. Кожен новий виклик функції отримує кадр з меншою адресою: main(…ca c4) → foo(…ca94, -48) → bar(…ca74, -32) → bar_is_now_closed(…ca54, -32). Різниця між main та foo більша (48 байт), бо main отримує аргументи argc і argv.

- Кадр стека (stack frame) містить: локальні змінні, збережені регістри та адресу повернення. Команда bt full показує вміст кожного кадру.

- gstack = обгортка над GDB. Обидва інструменти використовують механізм ptrace для читання стеку чужого процесу. gstack — швидко і без інтерактивності; GDB — повний контроль: навігація між кадрами, перегляд і зміна змінних.

- pause() — блокуючий системний виклик. Він зупинив процес у стані S+ (sleeping), що дозволило нам приєднатися до нього через GDB без гонки за часом.

### Завдання 2.5
Відомо, що при виклику процедур і поверненні з них процесор використовує стек.Чи можна в такій схемі обійтися без лічильника команд (IP), використовуючи замість нього вершину стека? Обґрунтуйте свою відповідь та наведіть приклади.

### Відповідь 
Ні, повністю замінити IP вершиною стека неможливо. 

IP — це регістр, який у кожен момент часу містить адресу наступної інструкції для виконання. Процесор автоматично збільшує його після кожної інструкції. Роль стека при викликах функцій.

Коли виконується call foo:
- Адреса повернення (поточний IP + розмір інструкції) кладеться на стек
- IP отримує адресу foo
- При ret — адреса знімається зі стека і записується в IP

Тобто стек зберігає адреси повернення, але IP все одно потрібен для виконання інструкцій між викликами.

### Функції:

- my_malloc(size) — first-fit пошук + розбиття блоку + sbrk() якщо нема вільних
- my_free(ptr) — звільнення + злиття сусідніх вільних блоків (coalesce)
- my_calloc(n, size) — malloc + memset(0)
- my_realloc(ptr, size) — якщо блок достатній, повертає той самий; інакше malloc + memcpy + free
- my_heap_dump() — діагностика: виводить всі блоки з адресами, розмірами та станом

Тести підтвердили:

- Вирівнювання по 8 байт (20 байт → 32, 8 байт → 32 мінімум)
- Злиття блоків: два сусідні вільні (32 + 104) злились у 168
- Повторне використання вільної пам'яті без виклику sbrk()
- Захист від double-free виводить попередження у stderr
